2.2 Как устроен Git

2.2.1 Первоначальная настройка

Перед началом работы с git необходимо указать имя пользователя и почту:

git config --global user.name "John Doe"

git config --global user.email "johndoe@example.com"

Для просмотра всех настроек выполним

git config --list --show-origin

2.2.2 Создание репозитория

Для работы с Git репозиторием нужно его создать или скачать.

  • Вы можете взять локальный каталог и превратить его в git репозиторий одной командой:

git init

  • Или вы можете скачать git репозиторий например с GitHub командой

git clone https://github.com/…

Мы воспользуемся первым способом.

После выполнения команды появляется каталог .git (о содержимом каталога поговорим позже). После этого все добавленные файлы в текущую директорию получат статус не отслеживаемые (untracked).

Это значит что git не будет запоминать эти файлы и не будет отслеживать их изменения.

Команда git add <file> сообщает git, что необходимо индексировать этот файл, то есть следить за его изменением.

Выполним наш первый коммит. 

git commit -am "first commit”
-a автоматически добавляет все изменения рабочей директории в индекс. Флаг применять осторожно, дабы не добавить лишнее
-m оставляем комментарий

При вводе git log мы сможем его увидеть.

А так же увидим хэш коммита, автора и дату изменения.

Но мы ничего не записывали в файлы. Запишем в “test1.txt” следующее:

книга по git:
https://git-scm.com/book/ru/v2/

При выполнении git status видим новое состояние modified, говорящее нам об изменении файла, но не добавлении его в следующий коммит.

Выполним git add test1.txt

Теперь выполним знакомую нам команду но уже с новым флагом:

git commit --amend

Далее пишем :wq и жмем Enter

Команда git commit --amend нужна если коммит выполнен слишком рано. Флаг позволяет внести изменение в уже сделанный коммит. Важно понимать что при выполнении этой команды предыдущий коммит просто перезаписывается, а значит вернуть его будет не возможно. Этот флаг следует применять осторожно. Например в избежании случаев “Ой, забыл добавить файл” или “Исправление грамматической ошибки”.

2.2.3 Ветка в Git

Ранее можно было заметить слово master, записанное в скобках рядом с нашим расположением в файловой системе. Помимо этого при выполнении git log можно увидеть запись “HEAD → master”.

Это указатели на эти ветки. Master (имя ветки можно сменить) является главной веткой в проекте. HEAD - это указатель, который показывает где мы сейчас находимся и какое состояние файлов мы сейчас наблюдаем.

Если мы хотим создать новую ветку, мы лишь создаем новый указатель. Сделать это можно командой git branch:

git branch testing

Как тогда Git узнает в какой ветке мы находимся? Для этого существует специальный указатель HEAD.

Для переключения на другую ветку, выполним команду git checkout <name>

На самом деле команда checkout позволяет переключаться не только между ветками, но и между коммитами. Существует ещё команда git switch master , позволяющая переключаться только между ветками.

Добавим еще один файл и выполним коммит

git commit -m “added new ‘test2.txt’ file”

Выполним команду git log

Новый коммит появляется над предыдущим.

Переключимся на ветку master. git checkout master

Создадим новый файл test3.txt

Проиндексируем и добавим в коммит:

git add . - этой командой можно индексировать все файлы в текущем каталоге

git commit -m “new file added”

Теперь если мы захотим посмотреть git log мы увидим только нашу ветку master.

Для просмотра и других веток выполним команду

git log --oneline --decorate --graph --all

2.2.4 .gitconfig

Как мы видим, появляются команды с большим количеством флагов и вводить их каждый раз не хочется. Для этого в Git существует .gitconfig

Хранится он в домашнем каталоге пользователя

На самом деле существует несколько файлов gitconfig:
1. На уровне системы (/etc/gitconfig).
git config --system

Для его редактирования должны быть права root.
2. Настройки пользователя хранит
~/.gitconfig или ~/.config/git/config . Используется при флаге --global

3. Для текущего git репозитория хранится в .git/config
Для запуска нужно указать флаг
--local

Внутри можем увидеть имя и почту. В конце файла добавляем alias

[alias]
  co = checkout
  ci = commit
  st = status
  br = branch
	graph = log --oneline --decorate --graph --all

Сохраняем и пробуем воспользоваться

Другой способ добавить alias git config --global alias.ci commit

2.2.5 .gitignore

.gitignore - файл внутри проекта, говорящий git какие файлы стоит игнорировать.

Некоторые файлы обычно не являются частью исходного кода проекта и не должны быть включены в коммит, так как они могут быть воссозданы из исходного кода или являются временными файлами, которые не нужны для дальнейшей работы. Файлы хранящие переменные окружения(.env) не должны быть включены в коммит. Игнорирование этих файлов позволяет уменьшить размер коммита и упростить процесс отслеживания изменений в проекте.

gitignore поддерживает шаблоны. К шаблонам применяются следующие правила:

  • Комментирование шаблонов (игнорирование). Нужно поставить символ “#”
  • Шаблоны применяются рекурсивно во всем дереве каталогов.
  • Для избегания рекурсии используйте символ слеш (/) в начале шаблона.
  • Для исключения каталога добавьте слеш (/) в конец шаблона.
  • Можно инвертировать шаблон, достаточно поставить символ “!” в начале правила.
Для команды git add существует флаг -f (или --force), который позволяет принудительно выполнить коммит для игнорируемого файла. Это полезно если под шаблон игнорирования попадает нужный файл.

Примеры шаблонов:
https://github.com/github/gitignore

Теперь посмотрим git status

Git игнорирует test.py , но .gitignore все же видит и сигнализирует нам что он не индексируется. Добавим и его в .gitignore

  1. Указатель HEAD указывает на ветку, принимающую результаты слияния.
  1. Проверить факт получения последних коммитов из удаленного репозитория (об удаленных репозиториях чуть позже).

Для слияния выполним команду git merge <name>, где <name> - ветка которую нужно слить в принимающую.

2.3.2 Отмена изменений

В Git существует универсальная команда для отмены изменений - git reset.

Вызывается она с одним следующих флагов:

  • --soft — оставляет файлы в текущем состоянии, но удаляет коммиты.

    git reset --soft HEAD~3 - удаляются последний три коммита.
  • --mixed — флаг по умолчанию (указывать не обязательно). Отменяет индексацию(git add) и отменяет коммит. При этом файлы будут изменены, и получат состояние modified.
  • --hard — сбрасывает изменения и удаляет все коммиты.

    git reset --hard ac12h2d - удаляются все коммиты, сделанные после ac12h2d
На самом деле git reset --hard не удаляет коммиты с диска, удаляются только ссылки на них в дереве изменений. Коммит можно будет вернуть, если на него указывает тэг.

git revert 234n7dc - создает новый коммит, отменяющий изменения в указанном коммите. Считается самым безопасным способом отмены изменений, особенно если над проектом работают несколько людей.

git restore test.txt - отмена изменений файла.
Так же restore позволяет отменить индексацию файла:
git restore --staged test.txt

2.3.3 Удаленный репозиторий

Удаленные репозитории представляют собой git репо, находящиеся на удаленной машине.

Клонируем репозиторий: git clone https://github.com/schacon/ticgit

Перейдем в каталог ticgit

git remote выводит название удаленного репозитория, origin дефолтное название которое дает git. При указании флага -v выводится ссылка.

При настройке нескольких удаленных репозиториев, если работают несколько пользователей, строк будет в разы больше. Это означает, что мы можем получить доступ до любого из этих пользователей.

Для того, чтобы присвоить репозиторию свое имя, достаточно выполнить следующую команду git remote add ticgit https://github.com/schacon/ticgit

Теперь если мы хотим получить изменения с удаленного репозитория, можно выполнить команду git fetch ticgit

Ветка master сейчас доступна под именем ticgit/master.

Стоит отметить, git fetch копирует данные в ваш локальный репозиторий, но не сливает с вашими наработками. При необходимости это нужно сделать самостоятельно.

Команда git push используется для создание резервной копии на сервере и отправки изменений для синхронизации кодовой базы.

Создадим репозиторий на github

На домашней страницу жмем кнопку new

Даем название и выбираем публичный или приватный репозиторий

Жмем Create repository и видим следующую картину:

Здесь нас интересует ссылка, копируем её.

Чтобы запушить наш репозиторий на гитхаб, нужно связать локальный и удаленный репозиторий. git remote add origin <скопированная ссылка с гитхаб>

origin - произвольное название удаленного репозитория.

Выполняем вход

С помощью git remote можно посмотреть подключенные удаленные репозитории

Запушим в гитхаб: git push origin master

master - ветка которую мы отправляем на удаленный сервер. Если ее там нет, то она будет создана.

Есть вариант слить ветку в другую. Для этого команда будет выглядеть так: git push origin master:remote_branch remote-branch - удаленная ветка в которую нужно слить.

Теперь в GitHub можем увидеть наш проект.

Если мы хотим запушить теги то нужно добавить флаг git push --tags

Если мы хотим запушить все, не ограничиваясь одной веткой: git push --all

git pull является сокращением двух других команд. Сначала команда выполняет git fetch , затем выполняется слияние содержимого. Для слитого содержимого выполняется новый коммит и указатель HEAD переносится на этот коммит.

git pull <remote-repo>

2.3.4 Теги в Git

Повсеместно встречается практика использования тегов для указания версии приложения. Теги устанавливаются на коммиты и являются своего рода указателем. Например:

git tag -a v1.0 -m "my version 1.0"

-m для установки сообщения “my version 1.0”

Удаление тега:

git tag -d v1.4
