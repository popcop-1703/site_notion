2. Практическая часть

2.1 Установка GitLab сервер

Установим зависимости

ставим postfix и выбираем “internet site”

Указываем почтовое имя

Изменим ссылку. Изначально на сайте указана gitlab-ee версия. на нужна комьюнити. соответственно gitlab-ce

Для начала сделаем подключение по http, позже это изменим.

При возникновении ошибок может помочь команда

sudo gitlab-ctl reconfigure

Меняем пароль в профиле администратора. Для этого жмем на иконку профиля → edit. После этого на боковой панели появится возможность смены пароля.

Заходим в раздел Projects и жмем “Создать проект”

Заполняем обязательные поля и создаем новый проект

2.2 Установка GitLab Runner

curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash

Для установки последней версии воспользуемся пакетным менеджером

sudo apt-get install gitlab-runner

Для регистрации ранера понадобится токен который можно получить на гитлабе. Заходим в наш проект.

Жмем settings на боковой панели, затем CI/CD

Затем в списке находим Runner и жмем Expand→ new project runner
Выберем ОС Linux и укажем тег “test”

Получаем токен:

Выполним команду:

gitlab-runner register

Указываем ip gitlab сервера, токен, имя, и указываем исполнителем shell.

Запускаем вручную

gitlab-runner run

Теперь ранер числиться в списке назначенных

2.3 Pipeline

В результате выполнения этого пайплайна мы развернем в тестовой среде простой одностраничный сайт.

Для начала предоставим права на использование sudo для gitlab-runner

Создадим первый pipeline.
Переходим на страницу проекта

Жмем на знак “+” и добавляем 2 файла

  1. Пишем название index.html и оставляем следующее содержимое
<!DOCTYPE html>
<html>

  <head>
    <title>Hello</title>
    <meta charset="utf-8">
  </head>

  <body>
      <h1>hello world!</h1>
  </body>
</html>

После выполняем commit.

  1. Для второго файла пишем название InstallWeb.sh и оставляем следующее содержимое
#!/bin/bash

Снова сохраняем

Cоздадим еще 1 новый файл названием “.gitlab-ci.yml”. Это будет наш первый pipeline

stages:              # Перечень этапов работ и порядок их выполнения
    - build
    - test

stage-build:                           # Название задачи, оно будет отображаться при просмотре всего пайплайна
    stage: build                       # Указывается этап, на котором будет выполняться задача
    tags:                              # здесь указывается runner, который будет выполнять ниже описанный скрипт
        - build
    script:
        - mkdir build
        - cp InstallWeb.sh build/
        - cd build
        - echo 'echo "Hello GitLab"' >> InstallWeb.sh
        - echo 'apt install apache2 -y' >> InstallWeb.sh
        - cat InstallWeb.sh
    artifacts:                             # Эти данные будут сохранены и в следующих stages мы их сможем использовать.
        paths:
            - build/
    allow_failure: true                     # Говорим, что ошибки возможны и их нужно игнорировать.

testing:
    stage: test
    tags:
        - test
    script:
        - sudo bash build/InstallWeb.sh 
        - sudo cp index.html /var/www/html/index.html

artifacts: stages являются автономными и не делят ресурсы друг с другом. Если вы хотите, чтобы файлы, созданные в одном stage, использовались в другом , сначала необходимо сохранить их как артефакты. Затем последующие stages смогут извлекать артефакты и использовать сгенерированные файлы.

Жмем commit changes

Переходим в раздел pipelines

Bидим выполнение нашего конвейера

Жмем на passed и увидим более подробное описание

Заходим в браузер, посмотреть что получилось
