2. Практическая часть

Есть различные готовые реализации кластера Kubernetes (K8S), например:

  • Minikube — готовый кластер, который разворачивается на один компьютер. Хороший способ познакомиться с Kubernetes.
  • Kubespray — набор Ansible ролей.
  • Готовые кластеры в облаке, например AWS, Google Cloud, Yandex Cloud и так далее.

Использовать одну из готовых реализаций — быстрый и надежный способ развертывания системы оркестрации контейнеров Docker. Однако, мы рассмотрим ручное создание кластера Kubernetes из 3-х нод — один мастер (управление) и две рабочие ноды (запуск контейнеров).

2.1 Системные требования

Для установки кластера Kubernetes каждый сервер должен соответствовать минимальным требованиям:

  • От 2-х процессоров
  • От 2-х Гб оперативной памяти
  • Уникальные MAC-адрес и Product_uuid
  • Сетевая доступность между нодами кластера

В нашем случае мы будем разоврачивать кластер k8s на трех нодах: одна мастер нода и две рабочие ноды.

2.2 Предварительная настройка системы

  1. Обновляем списки репозиториев и производим обновление всех пакетов в ОС
    apt update && apt -y upgrade
  1. Устанавливаем необходимые пакеты
    apt -y install apt-transport-https ca-certificates curl gnupg2 software-properties-common
    • apt-transport-https позволяет использовать HTTPS-транспорт для загрузки пакетов в систему при помощи APT
    • ca-certificates содержит набор сертификатов для проверки подлинности SSL соединений
    • curl — утилита для отправки GET, POST и других запросов на http-сервер
    • gnupg2 используется для безопасной работы с ключами и подписями
    • oftware-properties-common добавляет возможности для управления репозиториями ПО в системе
  1. Отключаем файл подкачки (SWAP). С ним Kubernetes работает плохо (поддержка включена только с версии 1.22 при условии корректной конфигурации)
    1. Проверяем включен ли он:
      free -h

      Если в строке с именем Swap нулевые значения, значит он не используется.

    1. Временное отключение файла подкачки
      swapoff -a

      После перезагрузки эта настройка прекратит свое действие

    1. Полное отключение файла подкачки

      Для перманентного отключения файла подкачки необходимо внести изменения в файл fstab

      nano /etc/fstab

      Найти строку, содержащую значение swap.img, и закомментировать ее
      Сохраняем и выходим из файла. Чтобы изменения были применены, необходимо перезапустить сервер при помощи команды 
      reboot:

      reboot
  1. Следующим шагом необходимо загрузить дополнительные сетевые модули из ядра операционной системы.
    1. Создаем конфигурационный файл с именем k8s.conf и прописываем в нем те сетевые модули, которые нам необходимы: overlay и br_netfilter
      cat <<EOF | tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF
    1. Загружаем ранее указанные модули из ядра операционной системы при помощи команд
      modprobe overlay
modprobe br_netfilter
    1. Проверяем, что сетевые модули были успешно загружены и активированы
      lsmod | egrep "br_netfilter|overlay"

      Если напротив значений overlay и br_netfilter во втором столбце присутствует цифра 0, то модули успешно активированы

  1. Так как большинство сетевых плагинов (CNI) в Kubernetes используют Iptables, а также маршрутизируют трафик через сетевой мост (bridge), нам необходимо активировать соответствующие сетевые параметры, которые отвечают за маршрутизацию трафика через сетевой мост
    cat <<EOF | tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF

    Перезапускаем параметры ядра:

    sysctl --system
  1. При использовании брандмауэра UFW (Uncomplicated Firewall) или любого другого firewall (например, Iptables) необходимо открыть следующие порты: 6443, 2379, 2380, 10250, 10259, 10257.

2.3 Установка CRI-O

Так как Kubernetes самостоятельно не запускает контейнеры, то ему необходим сторонний инструмент для таких задач — контейнерный движок. Движок для контейнеров (он же Container Runtime) предназначен для работы с контейнерами, включая инициализацию контейнеров, их запуск, остановку и т.д.

  1. Для начала задаем переменные, которые пригодятся нам для того, чтобы скачать необходимую версию Crio для определенной версии ОС. В переменной OS задаем имя и версию используемого дистрибутива, а в переменной CRIO_VERSION версию CRIO, которую будем скачивать:
    export OS=xUbuntu_22.04
export CRIO_VERSION=1.25
  1. Пакет crio присутствует в официальных репозиториях от дистрибутива openSUSE. Добавляем данные репозитории в систему:
    echo "deb https://download.opensuse.org/repositories/devel:/kubic:/libcontainers:/stable/$OS/ /"| tee /etc/apt/sources.list.d/devel:kubic:libcontainers:stable.list
    echo "deb http://download.opensuse.org/repositories/devel:/kubic:/libcontainers:/stable:/cri-o:/$CRIO_VERSION/$OS/ /"| tee /etc/apt/sources.list.d/devel:kubic:libcontainers:stable:cri-o:$CRIO_VERSION.list
  1. Импортируем GPG-ключи от репозиториев:
    curl -L https://download.opensuse.org/repositories/devel:kubic:libcontainers:stable:cri-o:$CRIO_VERSION/$OS/Release.key | apt-key add -
    curl -L https://download.opensuse.org/repositories/devel:/kubic:/libcontainers:/stable/$OS/Release.key | apt-key add -
  1. Обновляем списки репозиториев
    apt update
  1. Устанавливаем crio и дополнительные утилиты:
    apt -y install cri-o cri-o-runc cri-tools
  1. Запускаем crio и добавляем его в автозагрузку:
    systemctl start crio && systemctl enable crio
  1. Проверяем статус crio:
    systemctl status crio

    На этом установка CRI-O успешно завершена

2.4 Установка Kubernetes

  1. Добавляем GPG-ключ от репозитория Kubernetes:
    curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.29/deb/Release.key | gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg
  1. Подключаем официальный репозиторий Kubernetes:
    echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.29/deb/ /' | tee /etc/apt/sources.list.d/kubernetes.list
  1. Обновляем списки репозиториев:
    apt update
  1. Устанавливаем пакеты kubelet, kubeadm, kubectl:
    apt -y install kubelet kubeadm kubectl
  1. Выполняем операцию hold, чтобы при появлении новых версий для данных пакетов они не обновлялись 
    apt-mark hold kubelet kubeadm kubectl

2.5 Конфигурирование Kubernetes

2.5.1 Инициализация нод

  1. Инициализация мастер ноды.
    kubeadm init --pod-network-cidr=10.244.0.0/16

    С помощью флага --pod-network-cidr мы указываем подсеть, адреса которой будут назначаться подам в кластере. Подсеть 10.244.0.0/16 используется для работы flannel — специального плагина, который создает виртуальную сеть разработанную для работы с контейнерами.

    Инициализация мастер-ноды, в зависимости от скорости интернет-соединения, может занимать около 5 минут или даже больше. В начале процесса происходит загрузка необходимых компонентов kubernetes, таких как kube-apiserver, kube-controller-manager, kube-scheduler, kube-proxy, etcd, coredns. После этого выполняется первичная настройка, которая включает создание сертификатов безопасности, формирование конфигурационных файлов и запуск подов.

  1. После того как процесс инициализации мастер-ноды будет завершен (в терминале отобразится надпись «Your Kubernetes control-plane has initialized successfully!»), необходимо выполнить шаги, перечисленные в конце вывода.
    1. Создать в домашней директории скрытую папку с именем kube:
      mkdir -p $HOME/.kube
    1. Скопировать конфигурационный файл, содержащий данные для подключения к кластеру в ранее созданный каталог в домашней директории
      sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
    1. Назначить пользователя и группу на созданную директорию
      sudo chown $(id -u):$(id -g) $HOME/.kube/config
  1. Обратите внимание на самую последнюю строку в выводе команды (kubeadm join) — там будет сгенерирована команда, предназначенная для присоединения рабочих нод. Эту команду выполняем на оставшихся двух серверах. Команду kubeadm join необходимо выполнить от имени root-пользователя или от пользователя с правами sudo:
    kubeadm join 192.168.133.164:6443 --token cjedcm.mlb5rbhc578g263s \
        --discovery-token-ca-cert-hash sha256:91292017c3c036a72ce6bb202ac94cecf93e9b203b336f010daf32fc111a017d
  1. После того как оставшиеся узлы были успешно присоединены, возвращаемся на мастер-ноду и выводим список всех нод кластера при помощи команды:
    kubectl get nodes

    Как можно увидеть на скриншоте выше, наш кластер состоит из трех нод: одной мастер-ноды и двух рабочих нод.

  1. Также выведем список всех подов в кластере, используя команду:
    kubectl get po -A

2.5.2 Настройка сетевого плагина

Так как ранее мы задали подсеть, но не установили сетевой плагин flannel. Для полноценной настройки необходимо это сделать.

  1. Для установки flannel выполняем команду:
    kubectl apply -f https://github.com/coreos/flannel/raw/master/Documentation/kube-flannel.yml
  1. Проверяем статус запущенных подов
    kubectl get po -n kube-flannel

    Поды успешно запущены. Количество реплик flannel соответствует числу нод в кластере.

2.6 Проверка работоспособности кластера

2.6.1 Создание веб-сервера nginx

Для проверки рабоспособности поднимем веб-серввер nginx. Для этого необходимо создать yaml файл, который будет содержать конфигурацию нашего веб-сервера.

  1. Создадим файл nginx_deployment.yaml:
    nano nginx_deployment.yaml

    Запишем туда следующее:

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 2
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.16.1
        ports:
        - containerPort: 80

    Эти строки представляют конфигурацию объекта "Deployment" для запуска приложения Nginx в кластере Kubernetes. В них указывается следующая информация:

    • apiVersion: apps/v1 указывает на версию API, которая используется для определения структуры объекта.
    • kind: Deployment определяет тип объекта как "Deployment", который управляет развертыванием Pod'ов.
    • metadata: name: nginx-deployment задает имя развернутого объекта Deployment.
    • spec: selector: matchLabels: app: nginx определяет по каким меткам (label) выбирать Pod'ы для управления.
    • spec: replicas: 2 указывает количество реплик Pod'ов, которые нужно развернуть (в данном случае 2).
    • spec: template: metadata: labels: app: nginx определяет метку (label) для выбора Pod'ов.
    • spec: containers: name: nginx задает имя контейнера (Pod'а) - "nginx".
    • spec: containers: image: nginx:1.16.1 указывает на образ Docker, который необходимо использовать для контейнера (в данном случае Nginx версии 1.16.1).
    • spec: containers: ports: containerPort: 80 определяет порт, который контейнер будет использовать для входящего трафика (в данном случае 80)..

    Сохраняем изменения и выходим из файла

  1. Создаем наш deployment при помощи команды:
    kubectl apply -f nginx_deployment.yaml
  1. Проверяем статус созданного deployment:
    kubectl get po -n default

    Так как мы ранее задали две реплики приложения, то у нас запущено два пода с приложением.

2.6.2 Проверка nginx с помощью утилиты curl

Далее необходимо проверить, может ли Nginx отвечать на запросы.

  1. Для начала необходимо получить IP-адрес пода. Для этого воспользуемся командой kubectl describe, передав в качестве параметра имя одного из запущенных подов:
    kubectl describe pod nginx-deployment-848dd6cfb5-nv4bj

    Получаем объемную справку по состоянию пода, нам интересна информация об IP адресе, которая содержится в поле IP

  1. При помощи утилиты curl отправим запрос на IP-адрес пода:
    curl -i 10.244.1.2

    Как можно увидеть, Nginx успешно (код 200) отправил ответ на наш запрос.

2.6.3 Получения доступа к nginx из внешней сети

  1. Узнаем имя нашего созданного deployment:
    kubectl get deployment
  1. Для создания сопоставления, с помощью которого можно будет подключиться к контейнерам из внешней сети выполняется командой:
    kubectl expose deploy nginx-deployment --type=NodePort --port=80

    Данному deployment будет сопоставлен порт для внешнего подключения, который будет вести на внутренний (в нашем примере, 80)

  1. Узнаем внешний порт, который был сопоставлен внутреннему
    kubectl get services

    Видим наш deployment и в столбце Port(s) указаны сопоставленные порты в формате внутренний порт : внешний порт / протокол

  1. Узнаем IP-адрес ноды из нашего кластера (подойдет любая)
    ip a
  1. Подключаемся через браузер к этому IP-адресу и порту:
    192.168.133.165:30196

    Мы видим страницу nginx, что означает, что проброс портов выполнен верно.

2.7 Helm и Helm Charts

2.7.1 Установка helm

  1. Установим helm
    curl -fsSL -o get_helm.sh \
    https://raw.githubusercontent.com/helm/helm/master/scripts/get-helm-3
chmod 700 get_helm.sh
./get_helm.sh
  1. Проверим версию
    helm version

2.7.1 Добавление репозитория

  1. Проверим текущие репозитории
    helm repo list
  1. Добавим репозитории
    helm repo add stable https://charts.helm.sh/stable
    helm repo add bitnami https://charts.bitnami.com/bitnami
    • stable (в первом случае) и bitnami (во втором) это имя репозитория
    • https://charts.helm.sh/stable или https://charts.bitnami.com/bitnami это путь к чартам
  1. Проверим список репозиториев
    helm repo list

2.7.3 Поиск приложений в репозиториях Helm

  1. Найдем Wordpress
    helm search repo wordpress
