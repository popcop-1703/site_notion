2. Практическая часть

2.1 Настройка репликации

Для проведения данной работы нам потребуется две Linux-машинки. В рамках этой инструкции были развернуты два Debian 12. Рекомендуется установить две абсолютно одинаковые инсталляции OC.

Так как репликация у нас будет типа Master-Slave, то необходимо определиться с системой, которая будет выступать в качестве сервера “Master”. Выведем ее сетевые интерфейсы, чтобы в следующих шагах не запутаться в ip-адресах.

То же действие проделаем с подчиненным сервером

Далее устанавливаем СУБД на сервере:

sudo apt-get install postgresql postgresql-contrib

Аналогичное действие проделаем на сервере “Slave”

sudo apt-get install postgresql postgresql-contrib

После установки необходимо проверить, был ли запущен сервис СУБД на обоих серверах

service postgresql status
--------------------------
service postgresql start    #для запуска сервиса
--------------------------
service postgresql restart  #для перезапуска
--------------------------
service postgresql stop     #для остановки

Можем заранее посмотреть список баз данных на подчиненном сервере

sudo -u postgres -i
psql
\l

Те же действия повторим и с другим сервером

Теперь приступаем к настройке сервера “Master”.

Перейдем на УЗ пользователя postgres

sudo -u postgres -i

Под аккаунтом postgres необходимо создать пользователя для репликации

createuser --replication -P repl_user

Далее просмотрим расположение config-файла

psql -c 'SHOW config_file;'

Нам необходимо внести в этот файл некоторые правки

В файл необходимо внести следующие правки:

archive_mode = on               

archive_command = 'cp %p /oracle/pg_data/archive/%f' 
max_wal_senders = 10
wal_level = replica                     

wal_log_hints = on

Далее необходимо внести дополнения в файл pg_hba.conf, добавив имя пользователя для репликаций и IP-адрес подчиненного сервера.

Откроем конфигурационный файл pg_hba.conf с помощью текстового редактора nano

И внесем правки

host replication repl_user <IP_SLAVE_SERVER> scram-sha-256

После внесения правок в конфигурационные файлы, перезапустим сервис

На этом настройка сервера Master завершена. Теперь перейдем к настройке сервера Slave. Прежде всего правим файл postgresql.conf

Для внесения дальнейших изменений нам необходимо остановить сервер

systemctl stop postgresql
-------------------------
OR
-------------------------
service postgresql stop

Так как в режиме подчиненного сервера у нас все данные реплицируются с основного, нам необходимо удалить файлы из каталога main. Если у вас инсталляция не новая, можно предварительно сделать бэкап содержимого этого каталога.

Теперь проведем проверку работы процесса репликации. Для этого используем команду pg_basebackup с адресом основного сервера и именем пользователя для репликаций

pg_basebackup -R -h <IP_MASTER_SERVER> -U repl_user -D /var/lib/postgresql/15/main -P

Запустим сервис

Теперь просмотрим список реплицированных баз данных

2.3 Транзакции и точки сохранения

2.3.1 Примеры использования транзакций

В PostgreSQL транзакции могут быть управляемыми явным образом с помощью операторов BEGIN, COMMIT и ROLLBACK, либо они могут быть неявными, когда каждый запрос автоматически запускает новую транзакцию, если таковая не существует.

Предположим, что у нас есть задача обновить дату последней покупки для всех покупателей и затем удалить данные о покупателе с недопустимым форматом имени:

BEGIN;

-- Обновляем дату последней покупки для всех покупателей
UPDATE Покупатели
SET ПоследняяПокупка = CURRENT_DATE;

-- Удаляем данные о покупателе с недопустимым форматом имени
DELETE FROM Покупатели
WHERE Имя = 'Козлов';

COMMIT; -- Фиксация изменений

В результате у нас обновилась дата у 6 строк, и после выполнилось удаление одного из пользователей (Козлов).

В этом примере, если все операции внутри транзакции завершатся успешно, изменения будут зафиксированы. Если произойдет ошибка или нарушение целостности данных, транзакция будет отменена, и все изменения будут откачены.

2.3.2 Примеры использования точек сохранения

Предположим, что мы хотим выполнить ряд операций в рамках транзакции, но хотим иметь возможность откатиться только в случае ошибки при обновлении даты последней покупки для конкретного покупателя:

BEGIN;

-- Обновляем дату последней покупки для покупателя "Иванов Иван"
UPDATE Покупатели
SET ПоследняяПокупка = CURRENT_DATE
WHERE Имя = 'Иванов Иван';

-- Сохраняем точку сохранения перед обновлением следующего покупателя
SAVEPOINT update_next;

-- Обновляем дату последней покупки для покупателя "Петров Петр"
UPDATE Покупатели
SET ПоследняяПокупка = CURRENT_DATE
WHERE Имя = 'Петров Петр';

-- Если произошла ошибка, откатываемся к точке сохранения
ROLLBACK TO update_next;

COMMIT; -- Фиксация изменений

В этом примере, если обновление для покупателя "Петров Петр" завершится неудачно (например, из-за нарушения ограничений уникальности), мы откатимся к точке сохранения и не затронем изменения, сделанные для покупателя "Иванов Иван".
